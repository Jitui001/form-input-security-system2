# 前端表单输入安全校验系统

## 1. 项目背景与问题定义

### 问题分析
- **XSS攻击**: 表单输入未经净化，恶意脚本注入导致会话劫持、信息泄露
- **SQL注入**: 特殊字符绕过，数据库被攻击
- **CSRF攻击**: 非法表单提交窃取用户操作权限
- **数据泄露**: 敏感信息（身份证、电话）缺乏保护机制

### 现状痛点
- 前端缺乏有效的输入验证策略
- 用户输入直接发送到后端，校验责任不清
- 敏感字段无加密传输机制
- 无统一的安全审计日志

## 3. 威胁模型与风险分析

### 防护的攻击类型

| 攻击类型 | 示例 | 风险等级 | 防护方案 |
|--------|------|--------|--------|
| XSS | `<script>alert('xss')</script>` | 🔴 高 | 内容净化、编码、CSP |
| SQL注入 | `'; DROP TABLE users;--` | 🔴 高 | 特征检测、参数化查询 |
| CSRF | 伪造表单跨站请求 | 🟠 中 | Token校验、SameSite |
| 数据泄露 | 明文传输敏感信息 | 🔴 高 | 端到端加密(RSA+AES) |
| 暴力破解 | 密码字段连续错误输入 | 🟠 中 | 速率限制、验证码 |

### 攻击者模型
- **网络攻击者**: 中间人攻击、嗅探
- **恶意前端开发**: 植入恶意代码
- **社工攻击**: 诱导用户输入恶意数据

### 攻击路径
```
攻击者
  ├─ 网络嗅探 → 加密传输防护
  ├─ XSS注入 → 输入净化 + CSP
  ├─ 暴力破解 → 速率限制 + 验证码
  └─ CSRF伪造 → Token + SameSite
```

## 5. 核心安全设计

### 5.1 输入验证规则

```javascript
// 支持的验证规则类型
const validationRules = {
  email: /^[^\s@]+@[^\s@]+\.[^\s@]+$/,
  phone: /^1[3-9]\d{9}$/,
  idcard: /^\d{6}(18|19|20)?\d{2}(0[1-9]|1[0-2])(0[1-9]|[12]\d|3[01])\d{3}[\dXx]$/,
  username: /^[a-zA-Z0-9_-]{3,16}$/,
  password: /^(?=.*[a-z])(?=.*[A-Z])(?=.*\d)(?=.*[@$!%*?&])[A-Za-z\d@$!%*?&]{8,}$/,
  url: /^https?:\/\/[^\s/$.?#].[^\s]*$/i
};
```

### 5.2 XSS防护层

```javascript
// 多层防护机制
1. 黑名单检测: 检测 <script>, javascript:, onerror等
2. HTML实体编码: & → &amp; < → &lt; > → &gt;
3. 属性值白名单: 只允许特定安全属性
4. CSP头部: Content-Security-Policy: script-src 'self'
```

### 5.3 SQL注入防护

```javascript
// 危险字符检测
const sqlInjectionPatterns = [
  /(\b(UNION|SELECT|INSERT|DELETE|UPDATE|DROP|CREATE)\b)/i,
  /(--|;|\/\*|\*\/)/,
  /(\bOR\b.*=.*)/i
];
```

### 5.4 加密传输

```
敏感字段加密流程:
输入字段 → 选择敏感字段 → RSA公钥加密 → Base64编码 → HTTPS发送

后端解密流程:
HTTPS接收 → Base64解码 → RSA私钥解密 → 获得原始数据
```

### 5.5 CSRF防护

```
1. 服务器生成唯一Token → 返回前端
2. 前端表单自动填充Token字段
3. 提交时后端验证Token
4. Token一次性使用 + 自动刷新
```
